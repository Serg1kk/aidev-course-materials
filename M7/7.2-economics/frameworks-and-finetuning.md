# Framework landscape + экономика fine-tuning

> Срез: май 2026. Нет «лучшего» фреймворка — есть правильный под ваш контекст. Fine-tuning почти всегда дороже, чем кажется.

## 1. Framework landscape — два лагеря

| Лагерь | SDK / framework | Язык | Сильная сторона |
|---|---|---|---|
| **Provider-native** | Claude Agent SDK | Python, TS | глубже всех MCP + computer-use |
| | OpenAI Agents SDK | Python, TS | handoffs, tracing, встроенные guardrails |
| | Google ADK | Python, TS, Java, Go | A2A-протокол, Model Garden |
| **Independent** | LangGraph | Python, TS | stateful workflows, чекпоинты, human-in-the-loop. Самый мощный и сложный |
| | Mastra | TypeScript | TS-first, durable workflows |
| | Pydantic AI | Python | type-safe структурированные ответы |
| | CrewAI | Python | role-based мульти-агент, быстрый прототип |
| | Smolagents | Python | агенты-кодеры, минимальный слой |

## 2. Пять осей выбора (гоняйте задачу по ним)

| Ось | Вопрос себе | Куда тянет |
|---|---|---|
| 1. Язык / стек | На чём пишет команда? | Python → почти всё; TS → Mastra / ADK / OpenAI SDK |
| 2. Portability (lock-in) | Готовы залочиться на вендора? | Да → provider-native; Нет → independent |
| 3. Control-flow complexity | Линейная цепочка или граф с ветвлениями/состоянием? | Просто → CrewAI; Сложно → LangGraph |
| 4. Кастомизация | Нужно лезть внутрь шагов, прерывать? | Max → LangGraph / Pydantic AI; из коробки → CrewAI / SDK |
| 5. Экономика | Хотите свободно гонять между моделями под цену? | Да → independent + AI-gateway; Нет → SDK одного вендора |

> **Ключ:** оси 2 и 5 связаны. Provider-native SDK = удобство + привязка + меньше рычага по цене. Independent + gateway = свобода менять провайдера под бюджет, но больше сборки руками. Экономика — это **ось выбора** фреймворка, а не последствие выбора.
>
> **skill-based vs orchestrator** — открытый спор индустрии. Одни идут к десяткам узких «скиллов» (коррелирует с появлением Skills-подхода у вендоров), другие держат оркестратор (LangGraph) для сложного прода. Оба валидны — выбор по задаче, а не по моде.

## 3. Fine-tuning: когда оно вообще окупается

**Порядок решений (декаф-правило):** Prompt → RAG → Fine-tune → Distill. Fine-tuning — про **форму** (стиль, формат, тон, поведение), RAG — про **знания** (факты).

**Порог по объёму:**
| Дневной объём | Оптимальная стратегия |
|---|---|
| < 10K запросов/день | Prompt engineering |
| 10K-100K | Зона перехода; LoRA рациональна примерно с ~45K |
| > 100K | Fine-tune / distillation (кривые инвертируются) |

**Главный инсайт по стоимости:**
- Сам train-run стоит копейки (буквально единицы долларов).
- **96% трёхлетнего TCO fine-tuning** — это eval-сет, data curation, retraining и люди. Вы платите не за вычисления, а за качество данных.
- При большом устойчивом объёме fine-tuned небольшая модель может быть в разы дешевле, чем RAG поверх флагмана — но только если объём оправдывает построение и поддержку data/eval пайплайна.

**Техники (что в ходу в 2026):**
| Техника | Суть |
|---|---|
| LoRA | low-rank адаптеры (тренит 0.1-1% параметров); ранг 8-16 насыщает большинство задач |
| QLoRA | 4-bit база + адаптеры (70B на одной 80GB GPU; 7B на 24GB) |
| Unsloth | кастомные kernel'ы: быстрее + меньше VRAM |
| DPO | alignment без reward-модели (распространённый дефолт) |
| Multi-LoRA serving | 1 base + много адаптеров (vLLM/SGLang) → мульти-tenant |

> ⚠️ **Лицензионный риск:** многие провайдеры запрещают использовать их модели для дистилляции/обучения конкурирующих. Проверяйте лицензию ПЕРЕД дистилляцией с frontier-модели.
> **Контраргумент:** prompt-optimization методами (DSPy/GEPA) на ряде задач обгоняет RL-дообучение при кратно меньшем компьюте. Сначала выжмите промпт, дообучайте только когда упёрлись в потолок или нужен гарантированный формат.
