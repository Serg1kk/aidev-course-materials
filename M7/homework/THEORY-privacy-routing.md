# За рамками домашки: куда ещё утекают персданные и как это решают по-взрослому

> **Это теоретический блок «для размышления». В домашку он НЕ входит и не оценивается.**
> Цель: показать, что приватность в гибридном (локалка + облако) ассистенте — **сложная тема**, и
> простой роутер «нашёл PII в сообщении → локалка» это только верхушка. Чтобы реально закрыть утечку,
> думать надо на уровне **намерений → инструментов → агентов**, а не только входного текста чата.

---

## 1. Почему простого роутера недостаточно (главная мысль)

В домашке роутер смотрит PII **в тексте сообщения** юзера. Это правильный первый шаг, но утечка часто
происходит **уже ПОСЛЕ решения о маршруте**:

> Юзер пишет: «где мой заказ?» PII в сообщении нет → роутер отправил ход в **облако**.
> Облачная модель вызывает тул `getMyOrders()` → возвращаются имя, адрес, телефон, история заказов.
> Этот ответ тула подклеивается в контекст и уходит в облако. **PII утекла в облако после роутинга.**

Вывод: **PII попадает в контекст не только из чата.** Надо держать в голове ВСЕ источники, откуда
персданные могут «подгрузиться» в контекст модели:

- 🗄️ **База данных** (тулы `getMyOrders`, `getUserProfile`, любые запросы к таблицам users/orders).
- 🔎 **RAG / retrieval** (подтянутые чанки могут содержать персданные).
- 📄 **Файлы** (загруженные документы, вложения, скриншоты).
- 🧰 **Другие тулы / внешние API** (CRM, платёжки, тикет-системы).
- 🧠 **Память агента / история диалога** (предыдущие ходы с PII остаются в контексте).
- 🔢 **Эмбеддинги** (вектор персданных, отправленный в облачный embedding-эндпоинт).

**Поэтому защита приватности это не «фильтр на входе», а свойство всей архитектуры пайплайна.**
Один регэксп на чат закрывает 10% поверхности.

## 2. Способы решения (по нарастанию надёжности)

### Способ A. Маршрутизация по НАМЕРЕНИЮ / чувствительности тула (а не по PII в тексте)

Идея: решать local/cloud не по тексту сообщения, а по тому, **какие данные затронет операция**.

- **Тэгирование тулов.** У каждого тула метка чувствительности. Любой тул, который МОЖЕТ вернуть
  персданные (`getMyOrders`, `getUserProfile`, `getPayment`), помечен `PII_SOURCE: true`. Политика:
  «если для ответа понадобится PII_SOURCE-тул → форсим локалку».

  ```python
  TOOL_SENSITIVITY = {
      "searchProducts": "public",
      "getFaq":         "public",
      "getMyOrders":    "pii",     # адрес, история, платёж
      "getUserProfile": "pii",
  }
  def route(predicted_tool):
      return "local" if TOOL_SENSITIVITY.get(predicted_tool) == "pii" else "cloud"
  ```

- **Классификатор намерения** на маленькой модели/эмбеддингах (semantic-router, RouteLLM,
  vLLM Semantic Router): за <5-20 мс определяет тип операции и по нему выбирает ветку. Сигнал =
  предсказанный тул/тип задачи, а не сырой текст.

  > Грабли: тул, помеченный «public», но случайно возвращающий персданные (например `getOrderStatus`
  > тянет и адрес) обойдёт фильтр. Поэтому метки тулов надо ревьюить как код.

### Способ B. Маскирование / токенизация PII перед облаком

Если ход всё-таки идёт в облако: **обезличить** контекст перед отправкой и **восстановить** в ответе.

```
сырой контекст (PII) → Presidio Analyzer → Presidio Anonymizer
   → <PERSON_1>, <EMAIL_1>, <ORDER_1> → облачная модель
   → ответ с токенами → де-анонимизация по session-mapping → юзер видит реальные значения
```

- На практике: **LiteLLM proxy + Presidio guardrail** (`mode: pre_call`, `output_parse_pii: true`).
  Облако никогда не видит реальные значения; юзер получает их обратно.
- Обратимая (`encrypt` / instance-counter `<PERSON_1>`) vs необратимая (`hash`, `redact`) токенизация.

> **Пределы (честно):** (1) семантическая потеря — модель хуже рассуждает над замаскированными
> сущностями; (2) **mosaic / entity-linking атака** — по комбинации `<PERSON_1> живёт в <CITY_1> и
> работает в <ORG_1>` личность всё ещё вычислима; (3) кастомные форматы (внутренние order-id) Presidio
> не ловит без своих recognizer'ов. Маскирование необходимо, но не достаточно.

### Способ C. Разделение плоскостей данных (dual-LLM / минимизация)

Самый сильный подход: **приватные данные обрабатывает только локальная модель, облако видит максимум
обезличенное / минимизированное.**

- **Dual-LLM (S. Willison).** Привилегированная модель с тулами не видит сырой приватный контент;
  карантинная модель читает приватное, но без тулов; обмен через символьные переменные.
- **CaMeL (Google DeepMind).** Dual-LLM + capability-метки происхождения данных + ограниченный
  интерпретатор: значение с меткой «приватное / readers=alice» физически не может уйти туда, куда нельзя.
- **Минимизация данных (AirGapAgent-подход).** Локальная модель достаёт из БД сырой заказ, но в облако
  отдаёт ТОЛЬКО нужные поля (`{order_id, status, eta}`), а имя/адрес/платёж не покидают локалку.

  ```
  "где мой последний заказ?"
   → [локальная модель] getMyOrders() → сырой заказ (имя/адрес/...)
       извлекает ТОЛЬКО {order_id, status, eta}, остальное отбрасывает
   → [облачная модель] получает {order_id, status, eta}, формулирует ответ
   → "Заказ ORD-9912 отправлен, прибудет 3 июня."
  ```

### Способ D. Конфиденциальный инференс / Zero Data Retention (когда переархитектурить нельзя)

- **ZDR-режимы** провайдеров (OpenAI/Anthropic enterprise): запросы не сохраняются после ответа.
- **Confidential computing / TEE** (Azure AI Confidential Inferencing, AMD SEV-SNP + H100): промпт
  шифруется клиент→анклав, инфра провайдера не видит plaintext.

> Это контрактный/железный контроль: он мешает **хранению и наблюдению**, но модель всё равно
> *рассуждает* над приватными данными во время запроса. Это «по-взрослому для прода», часто overkill
> для пет-проекта.

## 3. Фреймворк принятия решения (на каждый ход)

```
1. Есть PII в самом сообщении юзера?           → ДА: только локалка (никакого облака)
2. Понадобится PII_SOURCE-тул (мои заказы/профиль)?
       → ДА, и хватает простой модели                → локалка
       → ДА, но нужно frontier-качество              → минимизация (C) или маскирование (B), потом облако
3. Это публичный/общий вопрос (каталог, FAQ)?  → облако
4. Неоднозначно?                                → классификатор намерения; при сомнении — локалка
```

| Запрос | PII в тексте | PII-тул | Нужно frontier | Решение |
|---|---|---|---|---|
| «найди синие кроссовки» | нет | нет | нет | облако |
| «политика возврата?» | нет | нет | нет | облако |
| «где мой заказ?» | нет | да | нет | **локалка** (хватит 8B) |
| «суммируй мои 5 заказов» | нет | да | средне | **локалка→облако (минимизация)** |
| «мой email X, отследи заказ» | да | да | нет | **локалка** |
| «посоветуй похожее на мои покупки» | нет | да | высоко | **маскирование→облако** или dual-LLM |

## 4. Какой моделью что делать

| Роль | Тир | Примеры | Заметка |
|---|---|---|---|
| Классификатор намерения / роутинг | tiny / эмбеддинг | all-MiniLM-L6 (22M), Qwen3-0.6B-1.7B | <5 мс, CPU, без GPU |
| Локальная модель (приватные данные, grounded Q&A над своими заказами) | 8B-14B | Qwen3-8B (Q6 для RU), Qwen3-14B, Qwen3-30B-A3B (MoE) | 8B хватает на ответы по своим данным |
| Облачная модель (сложное рассуждение, общие знания, качество) | frontier API | Claude / GPT / Gemini | при защите B/C сырой PII не видит |
| Детект PII (для маскирования) | rule+NER | Presidio + spaCy `en_core_web_lg` | локально, CPU |

## 5. Главное, что унести

1. **Приватность это свойство архитектуры, а не один фильтр.** Думай: намерение → какой тул → какой
   агент → откуда придут данные.
2. **PII втекает в контекст из многих мест** (БД, RAG, файлы, тулы, память), не только из чата.
3. **Надёжность защит по возрастанию:** роутинг по тексту < роутинг по намерению/тулу < маскирование <
   разделение плоскостей (dual-LLM/CaMeL/минимизация) < конфиденциальный инференс.
4. **Полного «бесплатного» решения нет** — это всегда баланс приватность / качество / латентность / сложность.
