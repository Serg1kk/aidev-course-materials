# Атаки на LLM/агентов, OWASP Top 10 и эшелоны защиты

> Срез: май 2026. Модель небезопасна by design: она не отличает инструкцию от данных. Защита вероятностна — «99% защиты = 0% защиты», атакующему хватает 1%.

## 0. Почему модель небезопасна by design

- Опасные знания «вшиты» в корпус обучения; alignment — тонкий слой поверх. Атаки эксплуатируют именно этот разрыв.
- Модель не отличает системную инструкцию от пользовательского ввода — всё попадает в одно окно внимания.
- **«You can't patch a brain»:** у prompt injection нет надёжного escape (в отличие от SQL injection, который детерминированно закрывается параметризованными запросами). Потолок защиты — 95-99%, не 100%.
- Эмпирика: исследования 2025 показывают, что при достаточном числе попыток уязвимы практически все модели; ~40% атак переносятся между поколениями моделей.

## 1. Каталог атак

### Обход alignment (jailbreak)
| Техника | Механика |
|---|---|
| DAN / «Mode» / персона | «игнорируй safety, отвечай как unrestricted-персона» |
| Grandmother / эмоц. отыгрыш | завернуть запрет в безобидный нарратив |
| Префикс-инъекция | «Начни ответ с „Конечно, вот…"» — внимание тянет к продолжению |
| Many-shot | сотни пар «плохой вопрос → плохой ответ» забивают большое окно (уязвимы long-context модели) |
| Crescendo | сдвигать модель к цели шаг за шагом |

### Обход guardrail / токенизатора (obfuscation)
| Техника | Механика |
|---|---|
| Base64 / кодировки | фильтр не читает, модель понимает |
| Layered bypass (input+output) | запрос и ответ в кодировке → пробивает И входной, И выходной фильтр |
| Невидимые / zero-width Unicode | токенизатор «сплющивает», модель получает чистый текст (обходит даже коммерческие prompt-shield системы) |
| TokenBreak / leetspeak | +1 символ меняет токенизацию; guardrail видит мусор, модель — смысл |

### Indirect injection (через внешний контент)
| Техника | Механика |
|---|---|
| README / коммент / письмо | вредная инструкция приходит из ДАННЫХ, агент её исполняет (например, агент-кодер читает README в CI/CD) |
| RAG-чанк | инъекция кладётся в вектор-базу через snippet → потом всплывает в контексте |
| **Lethal Trifecta** | недоверенные данные + приватные данные + канал наружу = эксплойт. Защита: **убрать любую одну «ногу»** |

### Атаки на данные / модель / инфраструктуру
Data poisoning · supply chain (отравленные веса, бэкдоры в публичных хабах) · DoS (шумная картинка ×10-100 compute, флуд токенами, `SELECT` без лимита) · model theft (выкачивание датасета массой API-запросов) · insecure output handling (доверие выдаче → RCE/XSS/DROP).

## 2. OWASP Top 10 for LLM (2025)

| ID | Риск | Пример из разработки | Частичная защита |
|---|---|---|---|
| LLM01 | Prompt Injection | indirect через README в CI/CD | разделение привилегий, фильтрация вывода, canary-токены, red-team |
| LLM02 | Sensitive Info Disclosure | запоминание чужого кода; RAG отдаёт данные не тому юзеру | RBAC на RAG-источники, классификаторы вывода |
| LLM03 | Supply Chain | отравленные веса, скомпрометированная зависимость | проверка origin, version pinning, SBOM, `--require-hashes` |
| LLM04 | Data & Model Poisoning | вредные данные в обучении/дообучении | проверка датасета |
| LLM05 | Improper Output Handling | XSS/RCE/DROP через text-to-SQL | zero-trust на вывод, sandbox/Docker для code-exec |
| LLM06 | Excessive Agency | агент с доступом ко всей БД | минимум прав, read-only, human-in-the-loop на необратимое |
| LLM07 | System Prompt Leakage | секреты в промпте утекают через jailbreak | только бизнес-логика+роль в промпте; секреты в ACL/RBAC |
| LLM08 | Vector & Embedding Weaknesses | RAG-чанк с секретом уходит наружу | фильтрация чанков, RBAC на retrieval |
| LLM09 | Misinformation | небезопасный сгенерированный код; галлюцинации с последствиями | SAST/CI на код, fine-tune под домен |
| LLM10 | Unbounded Consumption | DoS, неограниченное потребление токенов | лимиты/квоты, таймауты, мониторинг |

> Рядом с OWASP: **MITRE ATLAS** (матрица тактик в стиле ATT&CK), **NIST AI 100-2** (Adversarial ML Taxonomy). Готовится отдельный OWASP Top 10 для агентных приложений. OWASP — community-документ: используйте как карту, не как догму.

## 3. Реальные инциденты → урок

| Инцидент | Что случилось | Урок |
|---|---|---|
| Notion zero-click (PDF) | PDF с инъекцией на доске → агент собрал базу знаний и отправил на чужой адрес, без клика | lethal trifecta — режь ногу |
| Сервис-агент продал товар за $1 | уговорили согласиться на абсурдную цену — валидная оферта от лица компании | context minimization (read-only доступ к таблице цен) |
| Дипфейк «CFO» в видеозвонке (~$20M) | сотрудник перевёл деньги по реквизитам «руководителя» (голос+лицо склонированы) | out-of-band подтверждение транзакций |
| RAG-инъекция в корпоративном копайлоте | письмо → snippet в вектор-базу → фишинг-ссылка + «не указывай источник» | системный промпт нельзя «защитить как секрет»; данные = недоверенный ввод |
| Утечка ключей через code-exec | LLM-сгенерированный код исполнялся НЕ в sandbox | dockerize untrusted code решает проблему |
| Агент удалил production-БД | проигнорировал множественные code-freeze инструкции, сфабриковал данные | dev/prod separation, sandbox, chat-only режим планирования |
| **Supply-chain атака на популярный LLM-gateway** (2026) | через скомпрометированную зависимость украли токен публикации → бэкдор крал облачные/SSH/LLM-ключи | **кейс года.** version pinning, `--require-hashes`, ротация ключей; middleware = новая attack surface |
| Малварь использует установленный CLI-агент | npm-пакет применяет установленный coding-CLI как оружие | твой агент-инструмент может быть обращён против тебя |

> **Сквозной тезис:** реальная опасность — не «модель скажет лишнее», а «модель пойдёт в API/БД и сделает необратимое». При этом большинство утечек ПД по-прежнему происходят из ТРАДИЦИОННЫХ брешей (открытая БД, пароль 123456), а не из атак на AI. Но supply-chain атаки на AI-инфраструктуру уже в проде.

## 4. Эшелоны защиты (Defense-in-Depth)

Канон: **ни один guardrail не bypass-proof.** Защита = «швейцарский сыр», много слоёв.

| Слой | Приём / класс инструмента | Ограничение |
|---|---|---|
| Pre-flight | secret-scanning (git-secrets / TruffleHog / detect-secrets); PII-detection (Presidio) | не видит код / описательные тексты |
| Runtime input | guard-модель на injection/PII/jailbreak (<30-50 ms) | обходится Unicode-инъекцией |
| Runtime (agent) | agent-firewall: prompt-guard + alignment-checks + code-shield | alignment-checks экспериментальны |
| Dialog / topical | rule-based rails, фильтрация RAG-чанков до модели | сложнее настройка |
| Output validation | валидаторы вывода (JSON/SQL/PII/hallucination) | слабый input-layer |
| System-prompt hardening | защитная инструкция в промпте (дёшево, первый слой; снижает успешные атаки ~80%→5%) | надо вставлять в каждый бот |
| Red-team (offline) | «nmap для LLM» — фаззеры атак, встроенные в CI/CD | тестирование, не runtime |
| ❗ Network egress | контроль исходящих (ловит exfiltration кред через HTTP/MCP) | новый класс, незрелый |

> ⚠️ **inference-guardrail ≠ network-firewall.** Агент может пройти guard и слить креды следующим HTTP/MCP-запросом — нужен отдельный egress-контроль. Guard-модель сама является AI → тоже обходима. Bedrock/коммерческие guardrails сильны на английском (>90%), но заметно слабее на русском — для RU-аудитории это реальная боль.

### Архитектурные гарантии (детерминированные, не вероятностные)
- **Dual LLM / карантинная модель** (подход CaMeL, Google DeepMind): модель с инструментами НИКОГДА не видит недоверенный контент; его обрабатывает «безрукая» карантинная модель. Инъекция «заперта». Цена — падение автономности. **Безопасность ⇄ агентность:** чем безопаснее, тем «тупее» агент.
- **Plan-then-execute:** план строится только из доверенного ввода; результаты инструментов не меняют цепочку.
- **Статические правила на необратимое:** «нельзя `rm -rf`, если контекст загрязнён».
- **Главный продуктовый принцип — защищай ДЕЙСТВИЯ, не ОТВЕТЫ:** allow-list действий, параметризованный инструмент вместо свободного SQL, read-only грант на уровне БД, валидация + лимиты. Метрика защищённости — что можешь ограничить по действиям, а не «послушность модели».

## 5. Сколько вкладывать в безопасность (трезво)

Не каждому решению нужна тяжёлая защита: генератору постеров — нет; доступу к деньгам/ПД/транзакциям — да. Бюджет ИБ до ~30% стоимости внедрения там, где критично. Часто разумно ставить безопасность на 2-й этап — после доказательства ROI. Но для критичных данных безопасность — не «потом».
