# MCP vs CLI vs Skills — фреймворк выбора для AI-агентов, 2026

> **Аудитория:** разработчики, которые выбирают, как подключать внешние инструменты и рабочие процессы к AI-агентам и чат-ботам.
> **Цель:** дать чёткий фреймворк с опорой на данные — когда нужен MCP-сервер, CLI-инструмент, а когда Skill. Без «зависит от ситуации» без расшифровки.
> **Дата:** 2026-04-27. Цифры — из продакшн-бенчмарков, документации Anthropic и задокументированных кейсов.

---

## TL;DR — матрица решений на одну страницу

| Параметр | CLI | MCP | Skills |
|---|---|---|---|
| **Основной сценарий** | Coding-агенты с доступом к shell | Чат-боты, кросс-девайс, enterprise-аутентификация | Доменные знания, оркестровка рабочих процессов, документация |
| **Стоимость context window** | Низкая (1 Bash tool + lazy Skill) | Высокая (полная schema при старте сессии) | Минимальная (~50–100 токенов на Skill в context) |
| **Месячный расход (agent use)** | ~$3.20 | **~$55.20 (×17 дороже CLI)** | ~$4.50 |
| **Надёжность** | 100% (нет слоя протокола) | ~72% (28% тайм-аутов в тестах) | Skills самовосстанавливаются при поломке |
| **Аутентификация / разрешения** | Широкая (весь shell) | Гранулярная (OAuth 2.1 на инструмент) | Нет (нет живых подключений) |
| **Мультипользователь / enterprise audit** | Нет | Да | Нет |
| **Удалённый / кросс-девайс доступ** | Нет | Да | Нет |
| **Сложность настройки** | Средняя (shell + Skill) | Низкая (JSON-конфиг) | Низкая (Markdown-файл) |
| **Живые данные / текущее состояние** | Да (через CLI-команды) | Да (нативно) | Нет (статичные знания) |
| **Подходит для coding-агента** | Отлично | Хорошо | Отлично |
| **Подходит для чат-бота** | Плохо (требует shell) | Отлично | Хорошо |

**Правило выбора одной фразой:** используйте Skills для знаний и оркестровки рабочих процессов, CLI — для coding-агентов с доступом к файловой системе, MCP — когда нужна мультипользовательская аутентификация, удалённый доступ или централизованное enterprise-управление.

## Часть 1 — Определения

### MCP (Model Context Protocol)

MCP — открытый стандарт (первоначально опубликован Anthropic, сейчас принят в индустрии повсеместно), который определяет, как AI-агенты подключаются к внешним инструментам и источникам данных. MCP-сервер открывает набор типизированных tools, ресурсов и prompts через JSON-RPC протокол. Клиент (Claude Code, Cursor, Claude Desktop, ChatGPT) загружает сервер при старте сессии и передаёт инструменты модели.

**Ключевая характеристика:** schema каждого tool из каждого подключённого MCP-сервера загружается в context window при старте сессии. Это модель eager loading.

**Принят:** Anthropic, OpenAI, Google, Microsoft и фактически всеми AI-компаниями к апрелю 2026 года. В каталогах насчитывается свыше 10 000 MCP-серверов.

### CLI (Command Line Interface)

CLI — программа, которая принимает ввод и выдаёт результат через терминал. AI-агент вызывает CLI-инструменты через Bash/shell-доступ. Агенты вроде Claude Code, Codex CLI и Cursor уже имеют прямой доступ к терминалу — они запускают любой установленный CLI-инструмент без посредников.

**Ключевая характеристика:** агент пишет shell-команды в runtime. Это composable-подход (команды можно динамически объединять через pipe), а LLM-модели хорошо обучены на CLI-паттернах. CLI-интеграция требует одного универсального Bash tool в наборе агента — никакой загрузки schema на каждую функцию.

**Примеры:** `gh` (GitHub CLI), `supabase` CLI, Google Workspace CLI (gws), Stripe CLI, Ramp CLI.

### Skills (Claude Code Skills)

Skill — это Markdown-файл (`SKILL.md`) с YAML frontmatter, который описывает рабочий процесс, доменный паттерн знаний или набор инструкций. Агент загружает при старте сессии только frontmatter (имя, описание, trigger-фразы — ~50–100 токенов). Полное тело Skill загружается только тогда, когда модель решает его активировать.

**Ключевая характеристика:** lazy loading. Skill не держит живых подключений — это документация и оркестровка рабочих процессов, а не runtime-интеграция. Именно поэтому Skills дёшевы: 200 Skills потребляют примерно столько же context, сколько 3 MCP-сервера.

**Не подходит для:** всего, что требует запроса текущего состояния, выполнения команд или доступа к живым данным. В таких случаях используйте Skill в связке с CLI-инструментом.

## Часть 2 — Бенчмарк ScaleKit: разница в стоимости ×17

Наиболее полное независимое сравнение затрат по трём подходам опубликовал themenonlab.blog ([Prahlad Menon, март 2026](https://themenonlab.blog/blog/skills-vs-mcp-token-efficiency-ai-agents)):

| Конфигурация | Месячный расход агента |
|---|---|
| Только CLI | **$3.20** |
| CLI + Skills | **$4.50** |
| MCP (Direct) | **$55.20** |

**MCP в 17 раз дороже CLI при эквивалентной функциональности в персональном или small-team контексте.**

Ещё два показателя из того же исследования:
- **Надёжность MCP:** 28% тайм-аутов в тестах.
- **Надёжность CLI:** 100% — нет слоя протокола, нечему ломаться.
- **Самовосстановление Skills:** когда CLI-инструмент, на который ссылается Skill, ломается, агент может диагностировать и починить это. MCP-серверы меняют API без предупреждения, что приводит к тихим сбоям интеграций.

**Почему MCP дороже:** при каждом взаимодействии полные schemas инструментов находятся в context window. Каждый токен в context стоит денег при inference. Компактный CLI-набор использует один `bash` tool, тогда как MCP-сервер для той же функциональности может добавлять от 13 000 до 46 000 токенов за сессию.

## Часть 3 — Критика MCP и ответ на неё

### Критика Geoffrey Huntley

В середине марта 2026 года два известных специалиста запустили дискуссию «MCP умер»:
- CTO Perplexity публично объявил об отказе от MCP внутри компании в пользу API и CLI — несмотря на то что Perplexity уже создала и выпустила свой официальный MCP-сервер.
- Gary Tan (CEO Y Combinator) добавил ретвит с фразой «MCP sucks» и показал, что 100 строк CLI-кода, написанных за 30 минут, дали результат, который он описал как «в 100 раз лучше».

Технические аргументы против MCP:
- Загрязнение context window — каждый MCP-инструмент загружает своё полное определение в context агента.
- Трение при аутентификации — каждый MCP-сервер управляет своей аутентификацией, создавая накладные расходы на настройку для каждого сервера.
- Архитектурное несоответствие — агенты, которые уже работают в терминале и пишут shell-команды, не нуждаются в слое JSON-RPC-трансляции между собой и инструментами.

**Geoffrey Huntley** (open-source contributor) сформулировал суть: coding-агенты с доступом к терминалу уже работают с CLI нативно. Добавлять MCP для тех же инструментов — значит добавлять лишний слой, который потребляет context и создаёт режимы отказа.

### Ответ Allen Hutchison и паттерн трёх интерфейсов

Контраргумент, сформулированный на основе отраслевых ответов, включая анализ Phil Whittaker:

> "Skills and MCPs aren't competing solutions. Skills excel at information delivery and adaptive context management. MCPs provide structured tool access." (Skills и MCPs не конкурируют. Skills отлично подходят для передачи информации и адаптивного управления context. MCPs обеспечивают структурированный доступ к инструментам.)

**Архитектурный паттерн трёх интерфейсов** описывает производственную реальность зрелых команд:

1. **CLI:** для coding-агентов, которым нужен прямой composable-доступ к инструментам в терминале. Агент пишет команду, объединяет несколько CLI-вызовов в цепочку, динамически использует флаги. Лучший выбор для: Claude Code, Codex CLI, Cursor, любого агента с доступом к shell.

2. **MCP:** для структурированных внешних интеграций, где важны OAuth-гранулярность, мультипользовательские audit trails, удалённый доступ и enterprise governance. Лучший выбор для: чат-ботов (Claude Desktop, Claude Co-work, ChatGPT), B2B SaaS, enterprise-развёртывания, любого сценария, где «нетехнический пользователь авторизует AI действовать от его имени».

3. **Skills:** для знаний о рабочих процессах и инструкций по оркестровке, которые не требуют живых подключений. Лучший выбор для: кодирования доменной экспертизы, многошаговых рабочих процессов, снижения затрат на prompt engineering в каждой сессии, lazy-загрузки детальных инструкций.

**Ринат Абдуллин** (llm_under_hood, опыт продакшн-развёртываний) формулирует прямо: для персональных агентов — MCP; для корпоративных и бизнес-систем с LLM-бэкендами — терминальные сессии на сервере, где агенты взаимодействуют через CLI вместо «шумного и нестабильного MCP».

Источник: [llm_under_hood Telegram](https://t.me/s/llm_under_hood/777), 2026-03-20.

## Часть 4 — Когда выигрывает CLI

Выбирайте CLI, когда:

**1. Агент имеет доступ к shell.** Coding-агенты (Claude Code, Codex CLI, Cursor, OpenCode) уже работают в терминале с доступом к файловой системе. Добавлять MCP-сервер для GitHub или Supabase, когда `gh` CLI и `supabase` CLI можно просто установить, — значит создавать накладные расходы без прироста функциональности.

**2. Composability важна.** CLI-команды можно динамически объединять через pipe. Агент пишет ровно ту комбинацию флагов и pipes, которая нужна для задачи. MCP-инструменты имеют фиксированные schemas — агент может вызывать их только так, как они спроектированы. JeredBlu: «Agents are better at writing code than calling tools in a specific order or format. A good CLI lets the agent chain commands and build its own workflow on the fly.» (Агенты лучше пишут код, чем вызывают инструменты в строгом порядке и формате. Хороший CLI позволяет агенту строить собственный рабочий процесс на лету.)

**3. Token budget ограничен.** Один Bash tool обрабатывает любую CLI-команду. Для сравнения: GitHub MCP-сервер потребляет 46 000 токенов (25% от 200K context window) при старте сессии.

**4. Интеграция для одного разработчика или небольшой команды.** Функции MCP для governance (OAuth-гранулярность, мультипользовательский audit, централизованное управление) создают накладные расходы, которые не нужны для персональной автоматизации.

**5. Целевая платформа имеет зрелый, хорошо задокументированный CLI.** GitHub (`gh`), Supabase CLI, Google Workspace CLI (gws), Stripe CLI, Ramp CLI — все хорошо задокументированы, знакомы LLM-моделям и активно поддерживаются.

**Пример с Playwright (JeredBlu, [2026-04-05](https://www.youtube.com/watch?v=DJSkyZIxVWE)):**

> "The Playwright MCP fills up the context window really fast, whereas the Playwright CLI, paired with a CLI Skill, does the same exact thing and takes a fraction of the context." (Playwright MCP быстро забивает context window, тогда как Playwright CLI в паре с CLI Skill делает то же самое и занимает ничтожную долю context.)

Playwright MCP: ~13 647 токенов при старте сессии. Playwright CLI + Skill: ~100 токенов (Skill frontmatter) + Bash tool.

## Часть 5 — Когда выигрывает MCP

Выбирайте MCP, когда:

**1. Среда агента — чат-бот, а не coding-агент.** Claude Desktop, Claude Co-work, ChatGPT не имеют доступа к терминалу. MCP — единственный способ дать им доступ к внешним инструментам. Если агент не работает в shell, CLI недоступен.

**2. Важна гранулярность аутентификации.** MCP поддерживает OAuth 2.1 с гранулярными разрешениями: только чтение счетов против полной записи в бухгалтерию, доступ к конкретному проекту Supabase против всех проектов. CLI-инструменты, как правило, наследуют полные учётные данные текущего пользователя — без тонкой гранулярности, если не писать кастомные обёртки.

JeredBlu: «In general, MCP does authentication a bit better. You can scope it and give it specific permissions. Whereas, a CLI generally has access to all your credentials, your whole computer.» (В целом MCP лучше справляется с аутентификацией. Можно задать scope и дать конкретные разрешения. CLI же обычно имеет доступ ко всем учётным данным и ко всему компьютеру.)

**3. Нужен удалённый или кросс-девайс доступ.** MCP поддерживает удалённые серверы. Удалённый Supabase MCP доступен с телефона через мобильное приложение Claude, а также одновременно из Claude Desktop, Claude Co-work и Claude Code. CLI требует установки на каждом устройстве.

**4. Enterprise governance для нескольких пользователей.** Когда несколько команд или агентов нуждаются в доступе к одной интеграции, MCP обеспечивает: централизованную конфигурацию, OAuth audit trails, гранулярность разрешений на уровне инструментов, RBAC. Индивидуальные CLI-настройки на каждого разработчика этого не дают.

**5. Постоянные подключения к stateful-системам.** Подключения к базам данных, потоки данных в реальном времени, системы, требующие session state, — всё это хорошо ложится на MCP, где поддерживается постоянное подключение.

**6. Целевая платформа — low-code-среда.** n8n, Flowise и аналогичные no-code/low-code платформы органично работают с MCP как слоем интеграции между AI-агентом и автоматизацией рабочих процессов.

**Пример с Supabase:** JeredBlu явно предпочитает Supabase MCP перед Supabase CLI для продакшна — из-за гранулярности аутентификации, изоляции проектов и удалённого доступа с устройств, не предназначенных для разработки.

**Эмпирическое правило от systemprompt.io:** «If the prompt includes words like `query`, `fetch`, or `check current state` → you probably need MCP. Otherwise → Skill.» (Если в prompt есть слова `query`, `fetch` или `check current state` — скорее всего, нужен MCP. В остальных случаях — Skill.)

## Часть 6 — Когда выигрывают Skills (lazy loading)

Выбирайте Skills, когда:

**1. Агенту нужны доменные знания или многошаговый рабочий процесс без живых данных.** Skill может кодировать «как проводить code review для этого проекта» или «как генерировать еженедельный дайджест из этих источников» — многошаговую оркестровку, которая иначе потребовала бы повторяющихся prompt-инструкций.

**2. Token budget — основное ограничение.** 200 Skills потребляют ~15 000 токенов при старте сессии. 5 MCP-серверов — ~55 000 токенов. Для агента, который выполняет десятки разных типов задач, Skills — масштабируемый выбор.

**3. Знания нужно часто обновлять.** Skills — это Markdown-файлы в git-репозитории. Их обновление не требует изменений API, перезапуска серверов, смены версий протокола.

**4. Нужны самовосстанавливающиеся интеграции.** Когда CLI-инструмент меняет интерфейс, Skill можно обновить, задокументировав новый паттерн — агент прочитает обновлённый Skill и адаптируется. Когда MCP-сервер меняет API, интеграции молча ломаются, пока schema не обновят.

**5. Рабочий процесс персональный или для небольшой команды.** Для отдельных разработчиков и небольших команд накладные расходы на хостинг и поддержку MCP-сервера редко оправданы, если Skill + CLI даёт тот же результат.

**Кейс Rakuten:** сокращение времени на 87,5% через Skills по сравнению с MCP при эквивалентном выполнении задач. Источник: [intuitionlabs.ai](http://intuitionlabs.ai/articles/claude-skills-vs-mcp).

**Простое правило (docs.bswen.com):** «Skills for yourself, MCP for the team.» (Skills — для себя, MCP — для команды.)

### Модель загрузки Skill подробнее

| Уровень | Содержимое | Когда загружается | Размер |
|---|---|---|---|
| Tier 1 | YAML frontmatter (имя + описание + triggers) | Всегда, в каждой сессии | ~50–100 токенов на Skill |
| Tier 2 | Полное тело Skill (пошаговые инструкции) | Когда модель активирует Skill | Рекомендуется ≤200 строк |
| Tier 3 | Материалы по ссылкам (скрипты, шаблоны, references) | Только при явном обращении в шаге | Не ограничено, загружается по запросу |

Источник: анализ архитектуры Claude Code Skills (Simon Scrapes); Armin Ronacher о lazy loading Skills.

## Часть 7 — Гибридный паттерн: Claude Code + Bash + MCP

Производственный паттерн — не «выбрать одно», а сочетать все три подхода:

**Архитектура Claude Code по умолчанию:**
- Bash tool обрабатывает все CLI-взаимодействия нативно.
- MCP-серверы обрабатывают внешние аутентифицированные интеграции (подключения к базам данных, облачные API).
- Skills обрабатывают доменные знания, оркестровку рабочих процессов и CLI-документацию.

**Конкретный пример:**
```
Задача: "Проверь эти GitHub PR и создай Jira-тикеты для тех, что требуют follow-up"

Подход:
- GitHub: используем `gh` CLI (coding-агент имеет shell, нет дополнительного context-расхода)
  └─→ Пара: Skill "GitHub PR review workflow" (~75 токенов)
- Jira: используем Jira MCP (требует OAuth с проектным scope, не глобальные учётные данные)
  └─→ Описания инструментов загружаются один раз, работают на протяжении сессии
- Логика анализа: кодируется в Skill "PR-to-ticket decision"
  └─→ Загружается только когда шаг PR-review его активирует
```

Задокументированный рабочий процесс JeredBlu: использует remote MCP connectors (общие для Claude Desktop + Claude Co-work + Claude Code) для сервисов, где важны удалённый доступ и кросс-девайс шаринг, **плюс** CLI-инструменты в чисто coding-контексте для тех же сервисов при работе в Claude Code.

Главный вывод: **для одного сервиса можно использовать и MCP, и CLI** в разных контекстах. Выбор делается на уровне сессии — исходя из среды и того, что именно нужно от интеграции.

## Часть 8 — `mcp2cli` и утилиты автоконвертации

По мере развития дискуссии CLI vs MCP появились инструменты для конвертации между двумя подходами. Паттерн `mcp2cli` (обсуждался в сообществе с марта 2026 года) конвертирует определения MCP-сервера в CLI-совместимые обёртки, позволяя агентам с доступом к shell использовать те же интеграции через CLI-путь с меньшим context-расходом.

**Практическое применение:** если у вас есть MCP-сервер, который хорошо работает в контексте чат-бота (Claude Desktop), с помощью `mcp2cli` можно сгенерировать CLI-обёртку для использования той же интеграции в coding-агенте (Claude Code) за долю context-расхода. Вызовы API при этом идентичны — меняется только интерфейсный слой.

Это подкрепляет паттерн трёх интерфейсов: логика интеграции реализуется один раз, затем открывается через тот интерфейс, который подходит для конкретной среды агента.

**Ограничение:** автоконвертированные CLI наследуют качество описаний исходного MCP-сервера. Если описания MCP-инструментов были плохо написаны (общие глаголы, нет примеров, нет указаний «когда НЕ использовать»), CLI-обёртка унаследует те же проблемы. Конвертация не исправляет качество описаний.

## Часть 9 — Архитектурный паттерн трёх интерфейсов

Паттерн трёх интерфейсов описывает, как зрелые агентные системы в 2026 году структурируют интеграции с инструментами:

```
┌─────────────────────────────────────────────────────────────┐
│                    AI Agent System                           │
│                                                             │
│  ┌─────────────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │   Skills Layer  │  │   CLI Layer  │  │   MCP Layer   │  │
│  │                 │  │              │  │               │  │
│  │ Domain knowledge│  │ Shell tools  │  │ Authenticated │  │
│  │ Workflow rules  │  │ Composable   │  │ integrations  │  │
│  │ Lazy loaded     │  │ Terminal ops │  │ Remote access │  │
│  │ ~50 tok/skill   │  │ 1 Bash tool  │  │ Enterprise    │  │
│  └─────────────────┘  └──────────────┘  └───────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Правила для каждого слоя:**

| Слой | Что сюда относится | Что сюда НЕ относится |
|---|---|---|
| **Skills** | Оркестровка рабочих процессов, доменные знания, CLI-документация, фреймворки принятия решений | Запросы живых данных, проверки текущего состояния, аутентифицированные внешние вызовы |
| **CLI** | Любой инструмент с зрелым CLI, задачи coding-агентов, composable shell-операции | Кросс-девайс шаринг, интерфейсы для нетехнических пользователей, мультипользовательский auth-scope |
| **MCP** | Аутентифицированные внешние API, enterprise-интеграции, инструменты чат-ботов, удалённый/кросс-девайс доступ | Простые REST API, которые агент может вызвать напрямую через `curl`; инструменты без аутентификации |

**Когда REST API не требует аутентификации и прост:** пропустите MCP полностью. Пусть агент напишет вызов через `curl` или Python `requests`. Это «Programmatic Tool Calling» — расход токенов падает с 150 000 токенов (накладные расходы MCP schema в масштабе) примерно до **2 000 токенов** для типичной агентной API-задачи.

Источник: синтез NotebookLM из KB; [neuraldeep Telegram](https://t.me/s/neuraldeep/1987) — 100 MCP tools = 50K токенов; 100 CLI-команд = 1 Bash tool + описание Skill.

## Блок-схема выбора

```
СТАРТ: нужна внешняя интеграция
│
├── Есть ли у агента доступ к терминалу / shell?
│   ├── НЕТ → Среда чат-бота (Claude Desktop, ChatGPT, Co-work)
│   │         └─→ ИСПОЛЬЗОВАТЬ MCP (единственный способ получить доступ к инструментам)
│   │             └─→ Применить принципы дизайна MCP (5–15 tools, OAuth и т.д.)
│   │
│   └── ДА → Coding-агент (Claude Code, Codex CLI, Cursor)
│             │
│             ├── Требует ли интеграция:
│             │   - OAuth-гранулярность разрешений?
│             │   - Мультипользовательские audit trails?
│             │   - Удалённый / кросс-девайс доступ?
│             │   - Enterprise централизованное управление?
│             │
│             ├── ДА к любому → ИСПОЛЬЗОВАТЬ MCP
│             │               └─→ Держать количество tools ≤15
│             │                   └─→ Рассмотреть добавление соответствующего CLI Skill
│             │
│             └── НЕТ ко всем
│                 │
│                 ├── Есть ли у целевого сервиса зрелый CLI?
│                 │   ДА → ИСПОЛЬЗОВАТЬ CLI + Skill
│                 │         └─→ Один Skill документирует, как использовать CLI
│                 │             Bash tool обрабатывает выполнение
│                 │
│                 ├── Есть ли у целевого сервиса REST API без аутентификации?
│                 │   ДА → Пусть агент напишет вызов curl/requests напрямую
│                 │         (Programmatic Tool Calling — минимальный context-расход)
│                 │
│                 └── Интеграция — это знания о рабочем процессе / доменные правила
│                     без требований к живым данным
│                     → ИСПОЛЬЗОВАТЬ только Skill (без CLI, без MCP)

ПОСЛЕ ВЫБОРА:
│
├── Нужны ли рабочему процессу доменные знания или пошаговая оркестровка?
│   ДА → Добавить Skill вне зависимости от выбора CLI или MCP
│
└── Важен ли token budget?
    ДА → Посчитать суммарный MCP token cost при старте сессии
          Включить Tool Search (ENABLE_TOOL_SEARCH=true) при использовании MCP
          Заменить редко используемые MCP tools на Programmatic Tool Calling
```
