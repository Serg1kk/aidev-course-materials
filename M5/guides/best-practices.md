# Best Practices — Production-агенты в n8n (M5)

> Сводка production-правил для построения агентов в n8n. Distilled от практики 2025-2026.

---

## 1. Приоритет интеграции: нода > HTTP > MCP

Когда агенту нужен внешний сервис — выбирайте в строгом порядке:

### 1. Native node (первый выбор всегда)

Если в каталоге n8n есть нода для нужного сервиса (Google Calendar, Slack, Gmail, Notion, Airtable, Stripe, и сотни других) — используйте её. Это даёт четыре выигрыша одновременно:

- **Скорость** — прямой API-вызов без накладных расходов на discovery
- **Стабильность** — нода поддерживается командой n8n, OAuth настраивается в один клик
- **Отлаживаемость** — явные параметры в UI и понятные логи
- **Узкая область видимости** — агент видит ровно один инструмент с понятным контрактом

### 2. HTTP Request (когда нет нативной ноды)

Если сервис не покрыт нодой, но имеет REST API — берите HTTP Request. Конфигурацию можно импортировать прямо из curl-команды документации сервиса. Это всё ещё детерминированный вызов, его легко отлаживать, и архитектурно он проще MCP.

### 3. MCP Client Tool (только в трёх случаях)

- Сервиса нет ни в нодах, ни легко через HTTP, **но** есть готовый MCP-сервер
- Нужно динамическое обнаружение — агент сам выбирает между несколькими инструментами одного сервера, и набор инструментов может меняться без правки workflow
- Cross-instance оркестрация — один n8n-агент использует инструменты другого n8n-сервера или Claude Desktop

### Антипаттерн

> Делать кастомный MCP-сервер вокруг сервиса, для которого уже есть нативная нода. Получается медленнее, тяжелее в отладке, требует дополнительного сетевого слоя и не даёт никакого выигрыша в обнаруживаемости при фиксированном наборе инструментов.

**Аналогия:** native node — это приложение на телефоне (быстро, надёжно). HTTP — браузер (универсально, чуть медленнее). MCP — интернет-протокол: мощно для переменных тулов и cross-instance сценариев, избыточно для «дёрнуть один Slack».

## 2. Algorithm before AI (детерминированные guards)

LLM в production-системе должен быть **одним узлом** в жёстком детерминированном графе, а не центром принятия решений. Принцип: **дешёвые и быстрые проверки выполняются ДО дорогой LLM, и ПОСЛЕ — на её выходе**.

### Compound error

Цепочка из 5 шагов с точностью 90% на каждом даёт **59% успеха** в целом. Production-инциденты последних месяцев показывают реальные цены ошибки:

- Один автономный агент, не сумев накатить миграции, **молча решил полностью очистить базу данных** — с его точки зрения это привело структуру к ожидаемой. **1206 руковод. + 1196 компаний удалены**, агент сфабриковал данные о результатах для оператора (Replit, июль 2025)
- Другой агент вместо удаления кэш-папки выполнил `rmdir` в корне диска — путь параметра был относительным, и модель решила, что корень и есть рабочая директория

В обоих случаях **детерминированных guards не было**. Constraint жил только в промпте.

### 4 слоя guards в n8n

Подробно — в `algorithm-before-ai.md`. Кратко:

1. **Pre-agent guard (input)** — Switch / If нода до AI Agent проверяет валидность параметров
2. **Schema enforcement в MCP** — JSON Schema на параметрах инструмента (`enum`, `min`, `max`, `required`)
3. **Output guardrail** — Structured Output Parser + явная валидация после AI Agent
4. **HITL** — Wait-нода на критические действия (refund, удаление, full rollout)

### Главное правило

> **Constraint в коде = закон. Constraint в промпте = рекомендация.**

Это не риторика, а инженерное правило для production-агентов.

## 3. Tool Isolation

Атомарные тулы лучше монолитных. Разбивайте монолитный `manage_lead` на атомарные `search_leads` + `create_lead`. Это даёт:

- Точный повтор на упавшем фрагменте без пересжигания контекста
- Облегчение аудита — каждый вызов виден отдельно
- Меньше галлюцинаций при выборе параметров (узкий контракт = меньше степеней свободы)

### Anthropic 4 правила Tool Design

Из «Writing Effective Tools for AI Agents»:

1. **Пишите для модели, а не для человека** — positive instructions работают лучше negative («Сначала проверь бюджет через getBudget» вместо «Не вызывай без проверки бюджета»)
2. **Жёсткая JSON-схема + дефолты** — чем строже параметры (типы, enum, required), тем меньше галлюцинаций
3. **Ошибки как инструкции, не как stacktrace** — «Бюджет превышен, лимит 4000, предложи уменьшить» вместо `Something went wrong`
4. **20-tool ceiling** (Shopify Sidekick) — меньше = границы чётче. >50 = «смерть от тысячи инструкций»

### ACI (Agent-Computer Interface)

Терминология формализована в **SWE-agent paper** (Yang et al. NeurIPS 2024) — **+12.5% pass@1 на SWE-bench только редизайном интерфейса**, без изменения весов модели.

**Poka-yoke** (Toyota): структурно невозможно сделать неправильный вызов. Anthropic SWE-bench: relative paths → required absolute paths = entire failure class eliminated.

## 4. Memory выбор в n8n

### 5 типов памяти (n8n)

| Тип | Что | Когда |
|---|---|---|
| **Simple Memory** | История в памяти процесса | Демо, тесты. **НЕ работает в queue mode** |
| **Window Buffer Memory** | Последние N сообщений в скользящем окне | **Default для большинства агентов** |
| **Conversation Buffer Memory** | Полная история диалога | **Никогда в продакшен** — стоимость растёт линейно |
| **Postgres Chat Memory** | Сохранение через Postgres | Многопользовательские системы |
| **Vector Store Memory** | Эмбеддинги в Pinecone/Qdrant/Supabase | Семантическое извлечение, дороже в инфраструктуре |

### Рекомендации по умолчанию

- **Триаж и классификация** — Window Buffer Memory length=5
- **Разговорные ассистенты поддержки** — Window Buffer Memory length=20
- **Многопользовательские системы** — Postgres Chat Memory с user_id как ключом сессии
- **Долговременные агенты с базой знаний** — Vector Store Memory
- **Демо** — Conversation Buffer Memory; **в продакшен никогда**

### Реальный кейс перерасхода

Агент маршрутизатора с Conversation Buffer Memory на 500 запусков в день — **$400/мес** счёт OpenAI именно из-за того что Conversation Buffer отправляет в LLM всю историю сессии целиком на каждый запрос. Перевод на Window Buffer length=5 — **$40/мес**, в десять раз дешевле.

### Реальный баг продакшена (n8n issue #12958)

Настройка `Context Window Length` в Postgres Chat Memory **не уважается** частью узлов. По логам OpenAI улетает не последние N сообщений, а вся таблица истории — десятки килобайт текста, latency 4 сек → 1 мин, счёт растёт пропорционально.

> **Урок:** декларативные настройки в low-code требуют независимой проверки реального ввода в LLM через логи исполнения.

Подробно — в `memory-types-in-n8n.md`.

## 5. Лимит итераций (`maxIterations`)

Цикл агента может зациклиться, если инструмент возвращает противоречивые данные или промпт неустойчив. Верхняя граница на число итераций — **критическая защита** от неконтролируемого роста расходов.

### Рекомендации

- **Агенты первичной сортировки** — `maxIterations = 5`
- **Многошаговые исследовательские агенты** — `maxIterations = 10–15`
- **Долгие производственные пайплайны** — `maxIterations = 20`, не больше

Это не пессимизм, а защита от классических сценариев беглых агентов:

- Зацикливание на одной и той же ошибке
- Бесконечный поиск с уточнением запроса без прогресса
- Регрессия модели после обновления, которая ломает паттерн вызова инструмента

## 6. Structured Output

JSON output через **Structured Output Parser** в Agent-ноде вместо свободного текста. Для production — **обязательно**.

### Зачем

- Гарантия формата (next step не сломается на missing field)
- Возможность валидации (JSON Schema на выходе)
- Чёткая граница между «решение агента» и «формат для downstream»

### Как настроить

В n8n AI Agent ноде → подключить Structured Output Parser sub-node → описать ожидаемый JSON schema → агент возвращает ровно этот формат.

> Это место «тихих поломок production-агентов»: модель вернула что-то похожее на JSON, но с лишним полем — workflow ниже по течению ломается без понятной причины. Structured Output убирает этот класс ошибок.

## 7. GCAO для system prompt

**Goal / Context / Action / Output** — production-стандарт промпта для Agent-ноды.

```
Goal:    Измеримая, конкретная цель.
Context: Документы, требования, картинки, текущее состояние.
Action:  Глагол совершенного вида (напиши, создай, подготовь, вызови).
Output:  Формат вывода (XML, JSON, таблица, schema).
```

3 готовых шаблона под этот модуль — в `prompts/gcao-templates.md`.

## 8. Fallback-модель + retry

### Fallback-модель

В AI Agent можно указать запасную Chat Model на случай 5xx у основного провайдера. Типовой паттерн: при перебое OpenAI workflow автоматически уходит в Anthropic с чуть подправленным системным промптом.

### Retry с экспоненциальной задержкой

Сетевые сбои API провайдера — норма; повторные попытки с exponential backoff ставятся на ноду Chat Model или внешним обработчиком ошибок workflow.

## 9. Observability

Минимум для production-мониторинга:

- **Success rate** — доля успешно завершённых запусков workflow к общему числу
- **Latency на инструмент** — распределение времени выполнения каждого подключённого инструмента
- **Число итераций цикла** — если 90-й перцентиль внезапно подскочил с 3 до 7, это сигнал что промпт или набор инструментов деградировал
- **Токены и стоимость на запуск** — без этой метрики невозможно ловить инциденты со стоимостью
- **Доля fallback-вызовов** — сколько процентов запусков ушло во вторичную модель (индикатор состояния основного провайдера)

### Tools

- **Langfuse** — OSS, MIT, acquired by ClickHouse Jan 2026 ($15B). Подходит для self-host
- **LangSmith** — LangChain-native tracing
- **Arize Phoenix** — RAG-eval focus, 1T spans/month
- **n8n executions API** — встроенные логи + Error Trigger для уведомлений

## 10. Когда уходить из low-code

Признаки что задача переросла n8n:

- Агент использует **больше 4-5 инструментов** одновременно — отладка ломается, логи видны только у последнего узла
- **3+ уровня вложенности** sub-workflow — практически невозможно отлаживать
- System prompt уходит за **2000+ слов** — пора декомпозировать в код
- Нужна **sub-second задержка** или строгий контроль стоимости одного вызова
- Продукт — B2B SaaS с тысячами одновременных пользователей
- Регрессионные тесты в CI — canvas нельзя `pytest`-нуть

Это не недостаток n8n — это сигнал, что задача переросла инструмент. Уход в код — выбор, продиктованный предметной областью, а не модой.

### Типовая эволюция production-агента

```
Make / Zapier
    ↓ (быстрый прототип)
n8n или другой low-code
    ↓ (продакшен workflow с AI-вставками)
LangGraph / Pydantic AI / Mastra (масштаб, надёжность, наблюдаемость)
```

Не каждый проект доходит до конца — большинство останавливается на n8n, и это норма. Правильный инструмент — тот, который решает текущую задачу без оверкилла.
